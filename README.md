# Ascendra

An AI-powered startup simulation game. Build a technology company week by week — make product, hiring, and fundraising decisions while an AI-driven event engine throws realistic curveballs at you. Grow toward a $1M valuation, or run out of cash trying.

> **Status: v1 playable slice.** This is the first playable vertical slice, built to answer one question — *is the core loop fun?* — before investing in breadth or polish. Scope, rationale, and what's deliberately deferred are in [`company-docs/product-spec.md`](company-docs/product-spec.md). It is functionally complete and tested, at the internal-playtest stage (not yet publicly deployed).

## The core loop

Create a company → advance a week → the engine updates your metrics and may roll an **engineering event** → choose how to respond (each choice is a tradeoff) → watch cash, MRR, valuation, and technical debt move → raise money when you need it → reach the **$1M Success** or hit **Bankruptcy**.

Event narratives are generated by AI, with a deterministic templated fallback so the game keeps working even if the AI call fails, times out, or the backend is down.

## Tech stack

- **Frontend** — Next.js 15 (App Router), TypeScript, Tailwind CSS, Zustand, Recharts. All game state and the deterministic simulation engine live client-side (persisted to `localStorage`); no login, no database in v1.
- **Backend** — a stateless FastAPI service whose only job is to proxy AI event-narrative generation (OpenAI, `gpt-4o-mini`) so the API key never reaches the browser. No DB, no auth, no Redis.

## Getting started

Requires Node.js and Python **3.13** (pydantic-core has no 3.14 wheel yet).

### Frontend

```bash
cd frontend
npm install
npm run dev        # http://localhost:3000
```

The game is fully playable with no backend running — it uses the deterministic fallback for event narratives.

### Backend (optional for local play)

```bash
cd backend
python3.13 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload    # http://localhost:8000
```

With **no `OPENAI_API_KEY` set**, the backend runs in stub mode (a schema-valid canned response) — no OpenAI account needed for local dev. Set a real key in `.env` (see `.env.example`) to enable real AI-generated narratives.

### Full stack with Docker

```bash
cp .env.example .env        # optionally add a real OPENAI_API_KEY
docker compose up --build   # frontend :3000, backend :8000
./scripts/smoke.sh          # boots the stack and exercises the endpoints
```

## Running the tests

```bash
# Frontend — 82 tests
cd frontend && npm run typecheck && npm test

# Backend — 22 tests (OpenAI is mocked; no network calls)
cd backend && source .venv/bin/activate && pytest
```

## Deployment

Two deployables: the Next.js frontend (Vercel or the included Dockerfile) and the FastAPI proxy (any container host). Full instructions, env-var wiring, and the pre-deploy checklist are in [`company-docs/deployment.md`](company-docs/deployment.md).

> **Before putting a real `OPENAI_API_KEY` on a public URL:** the proxy has a built-in per-IP rate limit and body-size cap, but if you scale past one backend instance, front it with an edge/CDN rate limit for a true global cap. See [`company-docs/security-review.md`](company-docs/security-review.md).
