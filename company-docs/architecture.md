# Ascendra — Architecture Document (v1)

Status: Phase 0 deliverable. Scope is strictly the v1 feature set in `product-spec.md`. Anything not explicitly in that spec's scope is deferred per `roadmap.md` — see Section 8.

## 1. Overview & V1 Boundary

Ascendra v1 is a single-player, single-session, client-resident startup simulation. A player creates a company, then repeatedly advances a week; each week the client-side engine deterministically updates financial/customer/technical-debt state, occasionally rolls an Engineering event (narrative + 2-3 choices generated by AI with a deterministic fallback), and checks for the two v1 end states (Bankruptcy, $1M-valuation Success). There is no login, no cross-device save, and no server-side database in v1 — the only reason a backend exists at all is that the OpenAI API key cannot live in the browser, so event narrative generation needs one thin, stateless proxy endpoint.

**What v1 stands up:**
- Next.js 15 frontend (App Router, TS, Tailwind, Zustand, Recharts) holding all game state client-side, persisted to `localStorage`.
- One minimal backend service: a stateless FastAPI "AI event proxy" with a single real endpoint (`POST /v1/events/generate`) plus a health check. No database, no auth, no Redis, no ORM.
- A deterministic templated fallback library that ships in the frontend (or the backend — decision below) and can fully replace the AI call.

**What stays deferred (explicitly NOT built in v1):**
- Postgres, SQLAlchemy, Alembic, Redis, JWT auth — none of it. There is no persistent server-side entity to store yet (no accounts, no multi-save, no cross-session data).
- Employee Engine, Customer Engine (segments), Marketing/Sales pipeline, Competitor Engine, Fundraising Engine beyond Bootstrap/Angel/Seed, additional AI event categories (Customer/People/Investor/Market), AI board meetings, AI founder advisor, news system.
- Any notion of "user" as a backend concept.

This is a deliberate, sequenced trim of the original stack, not an abandonment of it — Section 8 defines exactly where each deferred piece reattaches in Phase 2/3 and why today's design doesn't block that.

## 2. System Components & Data Flow

### Components
1. **Frontend (Next.js 15 app, browser-resident)**
   - Owns all game state in a Zustand store, hydrated from and persisted to `localStorage` after every state-changing action (company creation, week advance, event choice, fundraising decision).
   - Runs the entire deterministic simulation engine client-side (pure TS functions — see Section 4). No simulation math happens on the server.
   - Calls the AI proxy only for event narrative generation, and only when an Engineering event roll succeeds for that week.
   - Owns the fallback template library and the decision of when to invoke it (client enforces the >5s timeout, not the server).

2. **Backend — AI Event Proxy (FastAPI, stateless)**
   - One real endpoint: `POST /v1/events/generate`. Takes a minimal game-state context, calls OpenAI, validates the response against a strict schema, returns it or a structured error.
   - No database connection, no session/auth, no per-user state. Every request is independent and self-contained (the client sends whatever context is needed).
   - Reason this exists at all: API key custody. Everything else about v1 could be a static site; this is the one exception.

3. **Deterministic fallback**
   - Lives in the frontend as plain TS (no network round-trip needed, so it's instant and free). The backend does not need its own copy since the frontend can trigger fallback locally without ever hearing back from the server — this also covers total network failure / proxy downtime, which a server-side fallback couldn't.

### Turn + Event Flow (per "Advance Week" click)

```
Player clicks "Advance Week"
        │
        ▼
[Frontend: Turn Engine — pure, deterministic]
  1. Apply revenue/expense/burn calc from current state
  2. Apply customer growth/churn calc
  3. Apply technical debt drift
  4. Recompute valuation
  5. Roll event probability (seeded RNG, weighted by technical debt)
        │
        ├─ no event rolled ─────────────────────────────┐
        │                                                │
        ▼ event rolled                                   │
  [Build event-generation request:                       │
   company name, industry, founder type, key metrics,    │
   technical debt level, week number]                    │
        │                                                │
        ▼                                                │
  POST /v1/events/generate  ──(timeout 5s)──►  FastAPI    │
        │                                        │        │
        │                              [call OpenAI,      │
        │                               validate schema]  │
        │                                        │        │
   ┌────┴──── success + valid schema ◄───────────┘        │
   │                                                       │
   │    ┌── failure / timeout>5s / malformed ──►[FRONTEND: pick fallback template
   │    │                                         matching event trigger + severity]
   ▼    ▼                                                  │
[Frontend: render event card with narrative + 2-3 choices] │
        │                                                  │
        ▼                                                  │
  Player selects a choice                                  │
        │                                                  │
        ▼                                                  │
[Apply choice's numeric consequences to cash/debt/customers]│
        │                                                  │
        ▼◄─────────────────────────────────────────────────┘
[Check end-state conditions: cash<=0 -> Bankruptcy;
 valuation>=$1M -> Success]
        │
        ▼
[Persist new full state + week's turn-history record to localStorage]
        │
        ▼
[Dashboard re-renders from updated Zustand store]
```

Turn processing (steps 1-5 minus the network hop) must complete well under the spec's 2-second budget; the network hop is bounded by the 5s timeout and only happens when an event actually rolls (4-10 times per 20-week game per the spec's ACs), which is what keeps this cheap.

## 3. Data Model (client-side state schema, v1)

All of this lives in one Zustand store, serialized as one JSON blob to a single `localStorage` key (e.g. `ascendra:save:v1`). There is exactly one save slot in v1 (no multi-save).

```ts
type Industry = "AI" | "Fintech" | "Ecommerce";

type FounderType =
  | "Engineer"
  | "ProductManager"
  | "SalesLeader"
  | "MarketingExpert"
  | "FinanceExecutive";

// Applied once at company creation, read (never mutated) by the turn engine every week.
interface FounderModifiers {
  featureDevSpeedMult: number;      // e.g. 1.15 = +15%
  technicalDebtAccrualMult: number; // e.g. 0.85 = -15%
  customerAcquisitionMult: number;
  fundraisingValuationMult: number;
  engineeringCostMult: number;
  // exactly the documented modifiers per founder type; each founder has >=2 nonzero entries
}

interface Company {
  name: string;
  industry: Industry;
  founderType: FounderType;
  founderModifiers: FounderModifiers;
  foundedAtWeek: 0;
}

// Current-state metrics the dashboard reads directly and the turn engine reads+writes weekly.
interface GameMetrics {
  week: number;                 // current week number, starts at 1
  cash: number;
  mrr: number;
  burnRate: number;             // weekly net burn
  runwayWeeks: number;          // derived: cash / burnRate (or Infinity if burnRate<=0)
  customerCount: number;
  teamSize: number;             // static/display-only in v1, not a live-managed lever
  technicalDebt: number;        // 0-100 scale (or similar), drives event probability/severity
  valuation: number;            // recomputed every week per formula in Section 4
  founderOwnershipPct: number;  // starts at 100, decreases on accepted fundraising offers
}

// One record appended per completed week; feeds the dashboard's 12-week trend chart and is the
// audit trail / debuggability surface called for by the spec's determinism ACs.
interface TurnHistoryRecord {
  week: number;
  metricsSnapshot: GameMetrics;      // full state at end of that week
  eventId?: string;                  // reference into eventLog, if an event fired this week
  fundraisingOfferId?: string;       // reference into fundraisingOffers, if raised this week
}

type EventTrigger = "engineering"; // only category in v1; union grows in Phase 2/3

interface EventChoice {
  id: string;
  label: string;
  description: string;               // the tradeoff text shown to the player
  consequences: {
    cashDelta?: number;
    technicalDebtDelta?: number;
    customerCountDelta?: number;
  };
}

interface EventLogRecord {
  id: string;
  week: number;
  trigger: EventTrigger;
  narrative: string;                 // AI-generated or fallback-template text
  source: "ai" | "fallback";         // which path produced this event, for QA/telemetry
  choices: EventChoice[];            // exactly 2 or 3
  chosenChoiceId: string | null;      // null until player picks
}

type FundraisingRoundType = "Bootstrap" | "Angel" | "Seed";

interface FundraisingOffer {
  id: string;
  week: number;
  roundType: FundraisingRoundType;
  offeredCash: number;
  impliedValuation: number;
  equityPct: number;
  status: "pending" | "accepted" | "declined";
}

// The single root object persisted to localStorage.
interface GameState {
  schemaVersion: 1;             // bump on any breaking shape change; enables migration logic later
  company: Company;
  metrics: GameMetrics;
  turnHistory: TurnHistoryRecord[];   // capped/pruned display-side to last 12 for the chart, but
                                       // full log kept for the session (small, session-only data)
  eventLog: EventLogRecord[];
  fundraisingOffers: FundraisingOffer[];
  gameStatus: "in_progress" | "bankrupt" | "success";
}
```

**Mapping to Phase 2 Postgres schema (migration path, for DBAdmin's future reference):**
- `Company` → `companies` table (add `user_id` FK once auth exists).
- `GameMetrics` (current values) → columns on `companies` or a `game_state` 1:1 table; historical values already live in `turn_history`.
- `TurnHistoryRecord` → `turn_history` table, FK to `companies`, one row per week — this is a direct lift, the shape doesn't need to change.
- `EventLogRecord` → `event_log` table, FK to `companies`; `trigger` becomes an enum column that simply gains more values (Customer/People/Investor/Market) in Phase 2/3 rather than changing shape.
- `FundraisingOffer` → `fundraising_offers` table, FK to `companies`; `roundType` enum gains Friends & Family/Series A/Series B values in Phase 2.
- `schemaVersion` on the client blob maps conceptually to an Alembic migration version once there's a real DB — keeping it from day one means the eventual "import local save into account" flow (if ever wanted) has a version to key off.

## 4. Simulation Engine Design

The engine is a set of pure TypeScript functions operating on `GameState`, with exactly one stochastic step (the event roll), isolated so it can be seeded/mocked in tests.

```
advanceWeek(state: GameState, rngSeed?: number): GameState
  = pipe(
      applyFinancialEngine,      // revenue, expenses, burn, cash, MRR
      applyCustomerGrowthChurn,  // simple growth/churn from product quality proxy + industry/founder mods
      applyTechnicalDebtDrift,   // baseline drift, modified by founder mult
      computeValuation,          // revenue-multiple formula, re-anchored by last accepted offer if any
      rollEngineeringEvent,      // the ONE stochastic step — takes rngSeed, weighted by technicalDebt
      checkEndStates             // bankruptcy / success
    )(state)
```

**Modules from the source spec relevant to v1:**
- **Financial Engine** — in scope, minimal: revenue = customers × price (fixed per industry or flat in v1 per AC #4 — pricing model itself is a single constant, not a pricing sub-system), expenses = fixed + salary-per-teamSize, burn = expenses − revenue.
- **Simulation Engine** (the orchestrator above) — in scope, this file *is* the v1 "Simulation Engine," just far smaller than the source spec's version.
- **Employee Engine, Customer Engine (segments), Fundraising Engine (as a full generated-investor-personality system), Competitor Engine** — deferred. v1's fundraising is a single formula-driven offer generator, not a full engine (see Section 6); no employee or customer-segment sub-systems exist.

**Valuation formula (v1):**
```
annualizedMRR = metrics.mrr * 12
baseValuation = annualizedMRR * REVENUE_MULTIPLE   // REVENUE_MULTIPLE is a tuned constant, e.g. 8-12x
valuation = lastAcceptedOffer
  ? max(baseValuation, lastAcceptedOffer.impliedValuation * GROWTH_DECAY_OR_CARRY_FACTOR)
  : baseValuation
```
Documented, single constant multiplier, no DCF/comps/competitor input — exactly matches AC #5 of Feature 2. The exact multiple and re-anchoring factor are tuning values Backend/Frontend can adjust without a schema change (they're constants, not stored state).

**Founder/industry modifiers:** Founder modifiers are numeric multipliers applied at the relevant calculation step (e.g., `technicalDebtAccrualMult` multiplies the drift amount before it's added). Industry in v1 affects only event/flavor content selection (per AC #4, starting stats are identical across industries) — so industry is read by the AI-context builder and the fallback-template selector, not by the financial/customer math. This cleanly sets up Phase 2's stated plan to let industry start affecting starting stats without restructuring the engine — industry is already a first-class field threaded through, it just isn't consulted by the math yet.

**Determinism:** Every function above is pure and takes explicit inputs (state, constants, and — only for the event roll — an RNG seed). This satisfies AC #2 of Feature 2 (deterministic given same state+inputs, no hidden non-determinism outside the event roll) and makes the whole engine unit-testable without mocking a backend.

## 5. AI Integration Layer

### Contract

**Request** (`POST /v1/events/generate`), sent by frontend when an event roll succeeds:
```json
{
  "trigger": "engineering",
  "context": {
    "companyName": "string",
    "industry": "AI | Fintech | Ecommerce",
    "founderType": "Engineer | ProductManager | SalesLeader | MarketingExpert | FinanceExecutive",
    "week": 7,
    "technicalDebt": 62,
    "cash": 184000,
    "mrr": 9200,
    "customerCount": 140,
    "severityHint": "moderate"
  }
}
```
Only the fields the AI needs for flavor go over the wire — no full game state, keeping payloads tiny and cheap, and reducing prompt-injection surface (nothing free-text from the player is ever included; company name is the only quasi-free-text field and is length/charset-restricted at creation per AC #1 of Feature 1).

**Response (strict schema, validated server-side before returning, and re-validated client-side defensively):**
```json
{
  "narrative": "string, 1-3 sentences",
  "choices": [
    { "label": "string", "description": "string", "consequences": { "cashDelta": -8000, "technicalDebtDelta": -15, "customerCountDelta": 0 } },
    { "label": "string", "description": "string", "consequences": { "cashDelta": -1500, "technicalDebtDelta": 5, "customerCountDelta": -3 } }
  ]
}
```
Constraints enforced by validation: `choices.length` in `{2,3}`, every choice has all three consequence fields present (numeric, finite, within sane bounds e.g. `|cashDelta| <= some cap tied to current cash/MRR` so the AI can't produce a game-breaking number), `narrative`/`label`/`description` non-empty strings under a max length. Any violation = treated identically to a failure (triggers fallback), per AC #4 of Feature 4.

### Fallback — exact trigger conditions
The frontend falls back to a pre-written templated event (same trigger category, severity-banded) when **any** of:
1. The proxy call errors (network failure, 5xx, proxy down).
2. The call exceeds a 5-second client-side timeout (matches AC #4 verbatim).
3. The response fails schema validation (missing fields, wrong choice count, non-numeric/out-of-bounds consequences, malformed JSON).

Fallback templates are a small static array/table in the frontend, keyed by `(industry, severityBand)`, each with pre-written narrative variants and the same choice/consequence shape as the AI response — so downstream consequence-application code has exactly one code path regardless of `source: "ai" | "fallback"`.

### Cheap-by-design measures
- AI is called only on the weeks an event actually rolls (spec bounds this to 4-10 times per 20-week game) — not every week.
- Request payload is minimal fields only, not full state — smaller prompt, cheaper call.
- No AI call for anything except Engineering-event narrative in v1: no advisor chat, no board meetings, no per-turn AI commentary.
- Fallback is a genuine substitute, not a degraded experience, so there's no pressure to retry the AI call — one attempt, one timeout, no retry loop (retries would multiply cost and latency risk for no gameplay benefit).
- (Optional, cheap, non-blocking enhancement Backend can add later: an in-memory or simple response cache keyed by `(industry, severityBand)` with a short TTL, so repeated similar contexts within a playtest session can reuse a recent narrative. Not required for v1 to hit "cheap," called out as an easy lever if cost becomes a concern.)

### Prompt-injection / basic output hardening
- Only `companyName` is player-influenced free text reaching the prompt; it's length- and charset-constrained at creation (1-40 chars, per AC #1) and should be inserted into the prompt via a clearly delimited template slot, not concatenated into an instruction-bearing string.
- The system prompt instructs the model to output only the JSON schema above (structured output / JSON mode if the provider supports it) — no execution of embedded instructions from context fields.
- Server-side schema validation is the real backstop, not prompt hygiene — treat every AI response as untrusted input regardless of prompt design, exactly as the fallback trigger conditions above already assume.

## 6. Interfaces / Contracts

### Frontend ↔ Backend API surface (v1 — this is the entire backend API)

`POST /v1/events/generate`
- Request/response shapes: Section 5.
- Errors: backend returns HTTP error (or a structured `{ "error": "timeout" | "invalid_response" | "upstream_error" }` body) — frontend treats any non-2xx or any validation failure identically, routing to fallback. Backend does not need to distinguish error types finely for the client's purposes in v1 (useful for backend logs, not for client branching).

`GET /v1/health`
- Trivial liveness check, no payload. Useful for DevOps monitoring even though there's nothing else to monitor yet.

That's the complete v1 backend surface. No endpoints for company/state/save/fundraising — those never leave the browser.

### Internal engine interfaces (frontend, for Backend/Frontend engineers to build against)
```ts
function advanceWeek(state: GameState, rngSeed?: number): GameState;
function computeValuation(metrics: GameMetrics, lastAcceptedOffer?: FundraisingOffer): number;
function rollEngineeringEvent(technicalDebt: number, rngSeed?: number): { fired: boolean; severity: "low"|"moderate"|"high" };
function generateEventNarrative(context: EventRequestContext): Promise<{ event: AIEventResponse; source: "ai" | "fallback" }>;
function applyEventChoice(state: GameState, event: EventLogRecord, choiceId: string): GameState;
function generateFundraisingOffer(roundType: FundraisingRoundType, metrics: GameMetrics, founderMods: FounderModifiers): FundraisingOffer;
function acceptOffer(state: GameState, offerId: string): GameState;
function declineOffer(state: GameState, offerId: string): GameState;
function checkEndStates(state: GameState): GameState["gameStatus"];
```
`generateEventNarrative` is the one function that owns the network call + timeout + fallback-selection logic internally, so the rest of the engine never has to know whether narrative came from AI or a template.

## 7. Key Decisions & Tradeoffs

1. **No backend database in v1, despite the original stack calling for Postgres/Redis/JWT.** Justification: v1 has no concept of "user," no cross-session persistence requirement, and no server-owned state at all — the turn engine and all state live and die with the browser session (persisted via `localStorage` for within-browser continuity). Introducing Postgres now would mean building an auth/account layer to make it meaningful, which is explicitly out of scope. Tradeoff: no cross-device play, no server-side backup of a save, acceptable per Resolved Decision #1 for an internal playtest.
2. **The backend exists solely as an AI-key proxy, not a general API.** This keeps the backend genuinely thin (FastAPI, no ORM, no DB driver, no auth middleware) — DevOps only needs to deploy a stateless service with one secret (the OpenAI key) and no data store to provision, back up, or migrate.
3. **Fallback logic lives client-side, not server-side.** This means fallback still works if the backend is fully down (deploy hiccup, DNS issue, etc.), not just if the AI call itself fails — a stronger reliability guarantee for AC #4 ("game must never block or break") than a server-side-only fallback would give.
4. **Valuation is a single documented constant-multiple formula, no re-implementation of DCF/comps.** Matches AC #5 exactly; keeps the "Success" end-state condition auditable/testable by QA (fixed test playthrough referenced in Dashboard AC #4).
5. **Fundraising in v1 is a single deterministic offer-generator function, not a "Fundraising Engine" with generated investor personalities.** The source spec's investor-personality/board-influence system is deferred; v1's `generateFundraisingOffer` takes MRR/customer count/founder mods and returns one offer — enough to satisfy the spec's ACs (valuation reflects MRR/traction, Angel from week 1, Seed gated on MRR threshold) without building the richer system prematurely.
6. **`schemaVersion` field included in client state from day one**, even though there's no migration tooling yet, specifically to avoid a Phase 2 headache if/when local saves need any transformation before a hypothetical "import into account" flow.
7. **Team size and product quality are static/derived numbers in v1** (per spec AC #4 and dashboard scope), read by the engine as constants/config rather than backed by real sub-systems — Frontend and Backend should treat them as pure display values in v1, not wire up any editing UI for them.

**What Frontend needs to know:** all game logic and persistence is client-side; the only network dependency is the one event-generation call, and it must be wrapped in a hard 5s timeout with the fallback path already fully built and tested before AI integration is considered "done" — the fallback path is not a lesser fallback path, it's a co-equal implementation per the spec's own risk #2.

**What Backend needs to know:** no ORM/DB setup needed for v1; the FastAPI service is stateless and horizontally trivial to deploy; the only real engineering effort is the OpenAI call, timeout handling, and strict response validation.

**What DBAdmin needs to know:** there is no schema to design in v1 — DBAdmin's first real task on this project is Phase 2 (employees, product/feature tracking), using Section 3's mapping table as the starting point for the Postgres schema.

**What DevOps needs to know:** v1 deploys as (a) a static/SSR Next.js frontend and (b) one stateless FastAPI container needing only an `OPENAI_API_KEY` secret and outbound HTTPS — no database, no Redis, no persistent volumes, no auth secrets. Deployment topology is intentionally minimal; don't provision Phase-2-sized infra now.

## 8. Phase-2 Seams (extension points, deliberately left in place)

- **`GameState.schemaVersion`** — enables a defined migration path once client saves may need transforming for a server-backed account model.
- **`EventTrigger` as a union type with one member** — adding Investor/People/Customer/Market categories in Phase 2/3 is a union extension plus new template/prompt content, not a shape change to `EventLogRecord`.
- **`FundraisingRoundType` as a union type with three members** — Friends & Family/Series A/Series B slot in the same way.
- **`Industry` union already threaded through the engine's context-building and AI/template selection, just not yet consulted by financial math** — Phase 2's "industry affects starting stats" plan doesn't require re-plumbing, just adding read sites in the financial/customer calc functions.
- **`teamSize` and product-quality proxy already exist as fields on `GameMetrics`**, just static/derived in v1 — Phase 2's Team Management and Product Management systems attach by making these fields *computed from* new sub-entities (employees, features/bugs) instead of constants, without renaming or restructuring `GameMetrics` itself.
- **The AI proxy's request contract (`trigger` + `context` object) is already generic across event categories** — Phase 2/3 categories reuse the same endpoint and validation pattern with a different `context` shape and prompt, not a new service.
- **The turn-history/event-log/offer-log pattern (append-only records keyed by week) is the same shape Postgres tables will take in Phase 2** — DBAdmin can lift the TS interfaces near-verbatim into table definitions (Section 3's mapping table).
- **The backend service boundary (stateless FastAPI, one job = AI proxying) is intentionally not merged into a monolith with future systems** — Phase 2/3's real backend (once auth/DB exist) can absorb this proxy as one route among many, or it can remain a separate microservice; either way today's design doesn't force a rewrite, just an extension.

## 9. Open Technical Questions

1. **Deterministic RNG seeding strategy**: should the event-roll seed be derived from `(week number + a per-game-session random seed generated at company creation)` so a given save is fully replayable/debuggable, or is per-call `Math.random()` acceptable for v1 given there's no replay requirement in the spec? Recommend the seeded approach (cheap to implement, meaningfully improves testability per AC #2) — flagging for Backend/Frontend to confirm during build rather than blocking Phase 0 on it.
2. **Exact `REVENUE_MULTIPLE` and technical-debt-to-event-rate tuning constants** are gameplay/balance decisions, not architecture — Product Owner and whoever plays the first internal builds should own tuning these against the spec's ACs (4-10 events per 20 weeks; runway <4 weeks = warning), not Architect.
3. **Where do fallback event templates get authored (content list)?** — this is a content task, not an architecture task; flagging so Product/Content assigns someone to write the per-industry/per-severity template bank before Frontend can fully wire the fallback path.
4. **Hosting choice for the two v1 deployables** (e.g., Vercel for frontend, a simple container host for the FastAPI proxy) is a DevOps call with no architectural dependency here — noting only so it isn't dropped, not proposing to resolve it in this doc.
